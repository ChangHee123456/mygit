## 펌웨어

펌웨어는(FirmWare)는 하드웨어 장치 내부에 저장되어 하드웨어를 직접 제어하고 동작시키는 저수준 소프트웨어를 의미한다. 하드웨어와 운영체제,응용 소프트웨어 사이에서 하드웨어의 기본 기능을 수행하며 일반적으로는 ROM이나 Flash등의 비휘발성 메모리에 저장되어 장치가 켜질 때 실행된다.

펌웨어의 특징은 여러가지가 있다.
 1. 하드웨어와 밀접한 연결
 일반적인 응용 프로그램처럼 다양한 컴퓨터에서 실행되는 것이 아닌 특정 하드웨어를 위해 제작된다. 예로는 키보드 내부에서 키 입력을 해석하고 USB 통신을 수행하는 펌웨어, 모터나 잉크 분사 장치를 제어하는 프린터 펌웨어 등이 있다.

 2. 비휘발성 메모리에 저장
 전원이 꺼져도 내용이 유지되어야 하기에 ROM, EEPROM, Flash Memory 같은 저장소를 사용한다.

 3. 하드웨어 초기화 및 제어 역할
 전원이 들어오면 가장 먼저 실행되어 CPU 초기화, 메모리 설정, 입출력 장치 설정, 통신 설정등의 작업을 하여 하드웨어를 준비한다.

 4. 변경이 어려우나 업데이트 가능
 과거에는 ROM에 저장되어 수정이 거의 불가능하였지만, 현재는 Flash 메모리를 사용하여 업데이트가 가능하다. 스마트폰 펌웨어나 공유기 펌웨어 업데이트가 그 예시라고 볼 수 있다.

## 펌웨어와 RTOS의 차이

펌웨어는 하드웨어를 동작시키기 위한 전체 프로그램이지만, RTOS는 펌웨어 내부에서 여러 작업을 효율적으로 관리하기 위한 운영체제이다. 따라서 RTOS는 펌웨어의 일부로 포함될 수 있다.

복잡한 시스템에서는 RTOS가 작업을 관리한다. RTOS가 센서 측정, 통신, 모터 제어 Task 등 각각의 작업에 CPU 시간을 배분한다.
펌웨어와 RTOS의 가장 큰 차이는 관리하는 것이 다르다는 것이다. 펌웨어의 경우 온도 센서 값 읽기나 LED 제어 등 장치의 기능 자체를 구현한다. 반면 RTOS의 경우는 여러 기능이 언제 실행될지를 관리한다. 예를 들어 센서 측정은 100ms마다 실행, 통신은 데이터가 들어오면 즉시 처리, 모터 제어는 1ms 주기로 실행 등 여러 작업을 어떤 순서와 시간에 실행할지를 담당한다고 볼 수 있다.

## 프로그램 간 우선순위가 필요한 이유

임베디드 시스템에서는 여러 작업을 동시에 수행하여야 하는 경우가 많다. 하지만 하나의 CPU 코어는 한 순간에 하나의 작업만 실행할 수 있기 때문에, 어떤 작업을 먼저 실행해야 할지를 결정해야 한다.
이때 사용하는 것이 작업 우선순위(Task Priority)이다. 예를 들어 자동차 제어 시스템에서는 브레이크 제어같은 우선순위가 높은 제어와 음악 재생같은 낮은 우선순위 제어 등이 있다. 이 둘의 제어가 동시에 요청된다면, 음악 재생보다 브레이크 제어가 먼저 실행되어야 한다.

그리고 우선순위가 필요한 이유는 여러가지가 있다.

 1. 실시간 응답 보장
 임베디드 시스템에서는 정해진 시간 안에 결과가 나오는 것이 중요하다. 가령 온도 표시같은 건 조금 늦어도 괜찮으나 모터 제어나 의료 장비등에서 늦게 응답하면 큰 문제가 발생할 수 있기 때문이다. 따라서 시간이 중요한 작업에는 높은 우선순위를 부여하게 된다.

 2. 중요한 작업이 CPU를 먼저 사용하도록 하기 위해
 CPU 자원은 제한되어 있기에 모든 작업이 같은 중요도를 가지면 중요한 작업이 늦어질 수 있기에 우선순위를 정해야 한다.

 3. 전력 관리
 저전력 임베디드 시스템에서는 CPU를 오래 켜놓으면 배터리가 빨리 소모된다. 여기에 우선순위를 활용하면 중요한 작업만 빨리 처리하고 CPU는 슬립 모드에 들어가 배터리를 절약할 수 있다.

우선순위 역전 현상은 높은 우선순위의 작업이 낮은 우선순위의 작업의 CPU 자원, 공유 자원을 점유로 인해 실행되지 못하는 현상을 말한다. 실제로 화성 탐사선 패스파인더에서 높은 우선순위 작업인 통신 처리가 낮은 우선순위 작업인 데이터 기록이 공유 자원을 잡고 있어 통신 작업이 지연되는 문제가 발생한 적이 있다. 이러한 문제를 해결하기 위해 여러가지 방법이 사용된다.

 1. 우선순위 상속
 높은 우선순위 Task가 기다리고 있으면, 자원을 가진 낮은 우선순위의 Task의 우선순위를 잠시 높여준다. 이를 통해 낮은 우선순위가 실행되고 자원을 반환하여 높은 우선순위 Task를 실행 할 수 있다.

 2. 우선순위 상한 프로토콜
 공유 자원을 사용하는 Task의 우선순위를 미리 높게 제한하는 방법이다. 예로 Sensor Buffer Lock 사용 시 최소 우선순위를 높음으로 설정하여 Lock을 잡는 순간 문제가 발생할 가능성을 줄일 수 있다.

 3. 공유 자원 최소화
 공유 자원을 줄이는 것으로도 우선순위 역전 현상을 완화할 수 있다. 전역 변수 공유 최소화나 데이터 복사 사용, 긴 작업 중에 Lock 유지 금지 등의 방법이 있다.

## Deadlock 현상

Deadlock(교착 상태)은 여러 작업(Task나 Process)이 서로가 가진 자원을 기다리면서 영원히 실행되지 못하는 상태를 말한다. 즉, 두 개 이상의 작업이 서로 점유하고 있는 자원을 요구하며 기다리기 때문에 어떤 작업도 진행할 수 없는 상태이다. 임베디드 시스템이나 RTOS에서는 주로 Mutex, Semaphore, 공유 메모리, 하드웨어 자원을 사용할 때 발생한다.

Deadlock은 아래 4가지 조건이 동시에 만족될 때 발생한다.

 1. 상호 배제(Mutual Exclusion) : 하나의 자원을 동시에 여러 작업이 사용할 수 없는 상태
 2. 점유와 대기(Hold and Wait) : 자원을 이미 가지고 있으면서 다른 자원을 기다리는 상태
 3. 비선점(No Preemption) : 다른 작업이 강제로 자원을 빼앗을 수 없는 상태
 4. 순환 대기(Circular Wait) : 작업들이 원형으로 서로 기다리는 상태

Deadlock을 해결하기 위해서는 여러 방법이 있다.

 1. Deadlock 예방 : Deadlock 발생 조건 중 하나를 제거
    자원 요청 순서 통일, 여러 자원 동시 요청, Lock 보유 시간 최소화

 2. Deadlock 회피 : 시스템이 자원 할당 전에 위험성 판단
 대표적으로 자원 요청 시 기능성을 검사하고 안전하면 허용하는 Banker's Algorithm이 있다.

 3. Deadlock 탐지 및 복구 : 이미 발생했다고 판단하고 해결하는 방법이다.
 일정 시간 이상 Lock 대기 감지 -> Task 재시작 -> 지원 강제 해체 -> 시스템 Reset

## 스케쥴러와 일반 프로그램의 차이

스케쥴러는 CPU를 어떤 작업에 할당할지 결정하는 운영체제의 핵심 구성 요소이다. 이러한 스케쥴러가 필요한 이유는 여러 작업을 동시에 처리하기 위해 매우 빠르게 Task를 전환하여 동시에 실행되는 것처럼 보이는 멀티테스킹을 할 수 있기 때문이고, 중요한 작업을 먼저 처리하거나 실시간 응답을 보장하기 위한것도 있다. 임베디드나 RTOS에서는 스케쥴러와 일반 프로그램의 차이가 특히 중요하다.

스케쥴러와 일반 프로그램의 차이는 다음과 같다.

            스케쥴러                일반 프로그램
목적        CPU 실행 순서 관리      특정 기능 수행
실행 주체   운영체제(RTOS/OS)       사용자 or 개발자가 작성
역할        Task 선택 및 전환       실제 작업 수행
CPU 권한    높은 권한(커널 영역)     제한된 권한
실행 방식   시스템 내부에서 동작      필요할 때 실행
예          RTOS Task Scheduler   센서 읽기 코드, 계산 프로그램
